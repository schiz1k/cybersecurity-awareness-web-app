# Глава 3. Реализация веб-приложения Cyber Arena

Документ подготовлен как расширенная техническая справка по проекту Cyber Arena. Его можно использовать как исходный материал для написания главы 3 дипломной/проектной работы или передать в ChatGPT, чтобы модель имела полный контекст о реализации веб-приложения.

Cyber Arena - русскоязычная веб-платформа для обучения кибербезопасности. Приложение объединяет интерактивные тренажеры, тесты, библиотеку учебных материалов, личный профиль пользователя, рейтинг участников и административную панель для управления контентом. Проект ориентирован на корпоративное обучение, внутреннюю аттестацию, регулярную практику специалистов по информационной безопасности и подготовку команд к соревнованиям.

## Общая характеристика реализации

Веб-приложение реализовано по клиент-серверной архитектуре. Клиентская часть отвечает за пользовательский интерфейс, маршрутизацию, отображение материалов, запуск тренажеров и работу с состоянием пользователя. Серверная часть предоставляет REST API, выполняет валидацию данных, работает с PostgreSQL и обеспечивает сохранение пользовательских результатов.

Стек проекта:

- frontend: React 19, TypeScript, Vite, React Router, Framer Motion, Lucide React;
- backend: Node.js, Express 5, TypeScript;
- база данных: PostgreSQL 16;
- валидация входных данных: Zod;
- подключение к БД: `pg`;
- запуск БД: Docker Compose;
- проверка качества кода: ESLint, TypeScript compiler;
- сборка frontend: Vite.

Основные каталоги проекта:

- `src` - клиентская часть приложения;
- `src/pages` - страницы приложения;
- `src/components` - переиспользуемые компоненты и игровые тренажеры;
- `src/hooks` - глобальные состояния и клиентские провайдеры;
- `src/utils` - утилиты для API, платформенного состояния и администрирования контента;
- `src/config` - конфигурация API и навигации;
- `src/data` - базовые демонстрационные данные;
- `server/src` - серверная часть приложения;
- `server/src/modules` - backend-модули по предметным областям;
- `server/src/db` - схема базы данных, seed-данные и подключение к PostgreSQL;
- `docs` - проектная, архитектурная и продуктовая документация.

Проект является одностраничным приложением. Маршруты работают через `HashRouter`, поэтому адреса имеют вид `/#/tests`, `/#/materials`, `/#/profile`. Такой подход упрощает размещение frontend на статическом хостинге, поскольку серверу не нужно отдельно обрабатывать fallback на `index.html` для каждого клиентского маршрута.

## 3.1. Нормализация базы данных веб-приложения

База данных приложения спроектирована в PostgreSQL. Схема хранится в файле `server/src/db/schema.sql`. Главная задача модели данных - разделить контент, пользователей, прогресс и результаты на независимые сущности, чтобы исключить избыточное хранение, упростить сопровождение и обеспечить целостность данных.

### Общая логика модели данных

В приложении есть несколько крупных предметных областей:

- пользователи и команды;
- пользовательские сессии;
- учебные материалы;
- прогресс изучения материалов;
- закладки;
- тесты и тренажеры;
- вопросы и варианты ответов;
- попытки прохождения;
- рейтинг.

Эти области не смешиваются в одной таблице. Например, материал хранится отдельно от своих акцентов и секций, тест хранится отдельно от вопросов и вариантов ответа, а пользователь хранится отдельно от попыток прохождения. Благодаря этому одна сущность не дублирует данные другой, а связи между сущностями описываются внешними ключами.

### Используемые enum-типы

В базе определены справочные enum-типы:

- `app_role` - роль пользователя: `student`, `instructor`, `admin`;
- `test_mode` - режим теста: `reflex`, `sequence`, `priority`, `quiz`;
- `content_status` - статус контента: `draft`, `playable`, `archived`;
- `difficulty_level` - уровень сложности: `base`, `medium`, `advanced`;
- `progress_status` - статус прогресса: `not_started`, `in_progress`, `completed`.

Использование enum ограничивает набор допустимых значений и снижает вероятность некорректных данных в базе. Например, в поле сложности нельзя случайно записать произвольную строку, не предусмотренную логикой приложения.

### Таблица `teams`

Таблица `teams` хранит команды, к которым могут относиться пользователи.

Поля:

- `id` - уникальный идентификатор команды UUID;
- `name` - название команды, уникальное значение;
- `city` - город команды;
- `created_at` - дата создания записи.

Связь: одна команда может включать множество пользователей. В таблице `users` для этого используется внешний ключ `team_id`.

### Таблица `users`

Таблица `users` хранит учетные записи пользователей.

Поля:

- `id` - уникальный идентификатор пользователя UUID;
- `email` - email пользователя, уникальное значение;
- `password_hash` - хеш пароля;
- `display_name` - отображаемое имя;
- `city` - город;
- `primary_track` - основное направление или роль обучения;
- `role` - роль пользователя в системе;
- `team_id` - ссылка на команду;
- `created_at` - дата создания;
- `updated_at` - дата последнего обновления.

Пароль не хранится в открытом виде. В таблицу попадает только хеш. Email приводится к нижнему регистру при создании и поиске пользователя.

### Таблица `user_sessions`

Таблица `user_sessions` хранит серверные сессии пользователей.

Поля:

- `token` - токен сессии, первичный ключ;
- `user_id` - пользователь, которому принадлежит сессия;
- `created_at` - время создания;
- `expires_at` - время истечения срока действия;
- `revoked_at` - время отзыва сессии, если пользователь вышел из аккаунта.

Сессии позволяют отделить факт входа пользователя от его учетной записи. При каждом защищенном запросе сервер проверяет, существует ли сессия, не истекла ли она и не была ли отозвана.

### Таблицы материалов

Учебные материалы разделены на несколько таблиц.

Таблица `materials` хранит основную карточку материала:

- `id`;
- `slug`;
- `title`;
- `category`;
- `level`;
- `read_time_minutes`;
- `read_time_label`;
- `summary`;
- `status`;
- `author_id`;
- `created_at`;
- `updated_at`.

Таблица `material_highlights` хранит краткие акценты материала. Один материал может иметь несколько акцентов. Порядок отображения задается через `sort_order`.

Таблица `material_sections` хранит тело материала. Материал разбивается на секции, каждая секция содержит заголовок и markdown-текст. Такой подход лучше, чем хранить все содержимое в одном большом поле, потому что секции можно сортировать, редактировать и расширять независимо.

### Таблицы закладок и прогресса

Таблица `material_bookmarks` реализует связь многие-ко-многим между пользователями и материалами. Один пользователь может сохранить много материалов, и один материал может быть сохранен многими пользователями. Первичный ключ составной: `user_id`, `material_id`.

Таблица `material_progress` хранит прогресс изучения. Она также связывает пользователя и материал, но дополнительно содержит:

- `status` - статус изучения;
- `completed_at` - дата завершения;
- `last_read_at` - дата последнего чтения.

Разделение закладок и прогресса позволяет независимо хранить два разных действия: пользователь может сохранить материал, но не завершить его, или завершить материал без добавления в закладки.

### Таблицы тестов и вопросов

Таблица `tests` хранит основную карточку теста или тренажера:

- `id`;
- `slug`;
- `title`;
- `headline`;
- `tag`;
- `category`;
- `description`;
- `difficulty`;
- `duration_minutes`;
- `duration_label`;
- `metric_label`;
- `accent_color`;
- `mode`;
- `status`;
- `config_json`;
- `author_id`;
- `created_at`;
- `updated_at`.

Таблица `test_benefits` хранит список преимуществ или учебных целей теста.

Таблица `test_deck_items` хранит дополнительные тезисы карточки теста, которые выводятся на странице.

Таблица `test_questions` хранит вопросы тестов. Она связана с `tests` через `test_id`.

Таблица `test_question_options` хранит варианты ответов для каждого вопроса. Для каждого варианта указывается текст, порядок отображения и признак правильности `is_correct`.

Такое разделение соответствует нормализованной структуре: вопрос не хранит варианты в одной строке, а каждый вариант является отдельной записью.

### Таблицы попыток прохождения

Таблица `test_attempts` хранит попытки прохождения тестов:

- `id`;
- `test_id`;
- `user_id`;
- `score`;
- `label`;
- `summary`;
- `tier_name`;
- `started_at`;
- `completed_at`;
- `meta_json`.

Одна попытка принадлежит одному пользователю и одному тесту. Пользователь может проходить один и тот же тест много раз. Это позволяет показывать историю попыток и выбирать лучший результат для рейтинга.

Таблица `attempt_answers` предназначена для хранения конкретных ответов в рамках попытки. Она связывает попытку, вопрос и выбранный вариант ответа. Сейчас основной сценарий сохраняет итоговую попытку, но структура базы уже позволяет расширить систему до подробного анализа ответов.

### Представление рейтинга

Для рейтинга создано SQL-представление `leaderboard_current`. Оно:

- берет пользователей из `users`;
- связывает их с командами из `teams`;
- выбирает лучший результат пользователя из `test_attempts`;
- вычисляет уровень пользователя по количеству баллов.

Логика уровней:

- от 940 баллов - `Легенда`;
- от 880 баллов - `Алмаз`;
- от 800 баллов - `Золото`;
- от 700 баллов - `Серебро`;
- ниже 700 баллов - `Бронза`.

Backend-модуль рейтинга дополнительно сортирует пользователей по лучшему результату и имени, добавляет порядковый номер через `row_number()` и ограничивает выдачу первыми 100 строками.

### Целостность данных

Целостность обеспечивается несколькими способами:

- UUID используются как первичные ключи;
- email пользователя уникален;
- slug теста и материала уникален;
- название команды уникально;
- внешние ключи связывают дочерние записи с родительскими;
- для дочерних таблиц используется `on delete cascade`, чтобы при удалении теста удалялись его вопросы, варианты, преимущества и попытки;
- для `author_id` и `team_id` используется `on delete set null`, чтобы удаление автора или команды не уничтожало весь контент;
- ограничения `check` не позволяют сохранить отрицательный score или неположительное время чтения/прохождения.

## 3.2. Реализация основного функционала веб-приложения

Основной функционал реализован на трех уровнях:

- пользовательский интерфейс на React;
- клиентское состояние и API-клиент;
- серверное REST API с PostgreSQL.

### Клиентская архитектура

Frontend находится в каталоге `src`. Центральные файлы:

- `src/main.tsx` - точка входа React-приложения;
- `src/app/AppRouter.tsx` - описание маршрутов;
- `src/app/AppProviders.tsx` - подключение глобальных провайдеров;
- `src/components/Layout.tsx` - общий layout, навигация и структура страниц;
- `src/config/navigation.ts` - список публичных пунктов меню;
- `src/config/api.ts` - адрес API и работа с session token;
- `src/utils/api.ts` - функции `apiGet` и `apiSend`.

Глобальные провайдеры подключаются в следующем порядке:

- `ThemeProvider` - тема интерфейса;
- `ContentProvider` - тесты и материалы;
- `LeaderboardProvider` - рейтинг;
- `PlatformProvider` - пользователь, профиль, попытки, закладки и прогресс.

Такой подход позволяет страницам получать нужные данные через хуки, не передавая их через длинные цепочки props.

### Маршрутизация

Маршруты описаны в `src/app/AppRouter.tsx`.

Основные маршруты:

- `/#/` - главная;
- `/#/tests` - каталог тестов;
- `/#/tests/:slug` - страница конкретного теста;
- `/#/leaderboard` - рейтинг;
- `/#/materials` - каталог материалов;
- `/#/materials/:slug` - страница конкретного материала;
- `/#/login` - вход;
- `/#/register` - регистрация;
- `/#/profile` - профиль;
- `/#/admin` - административная панель;
- неизвестные маршруты перенаправляются на главную.

Публичная навигация включает только основные пользовательские разделы:

- главная;
- тренировки;
- рейтинг;
- материалы;
- профиль.

Административная панель не включена в публичное меню. Даже если пользователь вручную откроет `/#/admin`, клиент проверит текущий host и перенаправит на главную, если адрес не является локальным.

### Работа с контентом

Контент приложения включает тесты и материалы. За его загрузку отвечает `src/hooks/useContentState.tsx`.

Логика работы:

1. При запуске приложения frontend пытается получить `/api/content/bundle`.
2. Если API доступно и возвращает данные, они становятся основным источником для страниц.
3. Если API недоступно или возвращает пустой набор, используется fallback из `src/data/siteData.ts`.
4. Административная панель использует методы `saveTest`, `deleteTest`, `saveMaterial`, `deleteMaterial`, `replaceAllContent`, `resetContent`.

На backend контент обрабатывает модуль `server/src/modules/content`.

Основные возможности:

- получить весь bundle тестов и материалов;
- получить список тестов;
- получить тест по slug;
- создать тест;
- обновить тест;
- удалить тест;
- получить список материалов;
- получить материал по slug;
- создать материал;
- обновить материал;
- удалить материал;
- заменить весь контент через импорт JSON.

В репозитории `content.repository.ts` данные собираются из нормализованных таблиц обратно в frontend-формат. Например, тест в базе разделен между `tests`, `test_benefits`, `test_deck_items`, `test_questions`, `test_question_options`, но frontend получает цельный объект `CyberTest`.

Для операций создания, обновления и массовой замены контента используются транзакции:

- `begin`;
- вставка или удаление связанных данных;
- `commit`;
- при ошибке `rollback`.

Это важно, потому что тест или материал состоит из нескольких таблиц. Если ошибка произойдет в середине операции, база не останется в частично обновленном состоянии.

### Реализация тестов и тренажеров

Тесты и тренажеры отображаются на странице `src/pages/TestDetailPage.tsx`. Страница получает `slug` из маршрута, ищет соответствующий тест в `ContentProvider`, отображает карточку и запускает нужный компонент в зависимости от `mode`.

Реализованные игровые компоненты:

- `ReflexDrill` - тренировка реакции;
- `SequenceDrill` - прохождение последовательностей;
- `PriorityDrill` - приоритизация задач;
- `QuizDrill` - тест с вопросами;
- `PasswordLab` - тренировка по надежности паролей;
- `PhishingLab` - распознавание фишинга;
- `PrivacyLab` - сценарии по приватности.

Формат теста задается полем `mode`. Если тест имеет статус `playable`, пользователь может пройти его. Если тест имеет статус `draft`, вместо тренажера отображается информационный блок о том, что сценарий появится позже.

После завершения тренажера вызывается `saveAttempt`. Результат передается в `PlatformProvider`, затем отправляется на сервер через маршрут:

```text
POST /api/users/current/tests/:testSlug/attempts
```

Сервер проверяет текущую сессию, находит пользователя и тест, сохраняет результат в `test_attempts`, вычисляет уровень по баллам и возвращает обновленное состояние пользователя.

### Работа с пользователями

Пользовательские функции реализованы через `src/hooks/usePlatformState.tsx` на клиенте и `server/src/modules/users` на сервере.

Реализованные пользовательские сценарии:

- регистрация;
- вход;
- выход;
- восстановление состояния по session token;
- обновление профиля;
- сохранение попытки прохождения теста;
- добавление или удаление материала из закладок;
- отметка материала как завершенного.

При регистрации пользователь отправляет:

- email;
- пароль;
- отображаемое имя;
- город;
- направление обучения.

Сервер:

1. Валидирует данные через Zod.
2. Проверяет уникальность email.
3. Хеширует пароль.
4. Создает пользователя.
5. Создает серверную сессию сроком на 30 дней.
6. Возвращает `sessionToken` и состояние пользователя.

При входе сервер находит пользователя по email, проверяет пароль и создает новую сессию.

Состояние пользователя `PlatformState` содержит:

- `accountEmail`;
- `learnerName`;
- `role`;
- список завершенных материалов;
- список закладок;
- историю попыток.

В frontend session token хранится в `localStorage` под ключом `cyber-arena-session-token`. При запросах к защищенным API он передается в заголовке:

```text
Authorization: Bearer <token>
```

### Работа с материалами

Каталог материалов отображает учебные записи из `ContentProvider`. Пользователь может открыть отдельный материал, прочитать его, сохранить в закладки или отметить как завершенный.

Основные действия:

- добавление/удаление закладки:

```text
POST /api/users/current/bookmarks/:materialSlug/toggle
```

- отметка материала как завершенного:

```text
POST /api/users/current/materials/:materialSlug/complete
```

Серверные операции используют текущую сессию пользователя. Для закладок применяется таблица `material_bookmarks`, для завершения материала - `material_progress`.

### Рейтинг

Рейтинг строится на основе результатов пользователей. Клиентский хук `useLeaderboardState.tsx` запрашивает текущий рейтинг через API.

Серверный маршрут:

```text
GET /api/leaderboard/current
```

Backend получает данные из SQL-представления `leaderboard_current`, сортирует их по лучшему результату и имени пользователя, затем возвращает список до 100 участников.

Рейтинг содержит:

- место;
- имя;
- команду;
- город;
- направление;
- уровень;
- баллы.

После сохранения новой попытки frontend вызывает обновление рейтинга, чтобы пользователь сразу видел влияние результата на таблицу лидеров.

## 3.3. Обеспечение информационной безопасности

В проекте реализован базовый набор мер информационной безопасности. Они закрывают основные риски учебного веб-приложения: хранение паролей, доступ к профилю, ограничение административных действий, валидация данных и контроль CORS.

### Хранение паролей

Пароли пользователей не сохраняются в открытом виде. Для новых учетных записей используется алгоритм `scrypt` из встроенного модуля Node.js `crypto`.

Логика:

1. При регистрации создается случайная соль длиной 16 байт.
2. Пароль и соль передаются в `crypto.scryptSync`.
3. Полученный хеш сохраняется в формате `scrypt:<salt>:<hash>`.
4. При входе пароль пользователя хешируется с той же солью.
5. Результаты сравниваются через `crypto.timingSafeEqual`.

Использование `timingSafeEqual` снижает риск атак, основанных на измерении времени сравнения строк.

В коде также есть fallback для старых пользователей с SHA-256-хешем. Он нужен только для совместимости с учетными записями, созданными до перехода на scrypt.

### Сессии и авторизация

После регистрации или входа сервер создает session token:

- токен генерируется через `crypto.randomBytes(32)`;
- токен сохраняется в таблице `user_sessions`;
- срок действия сессии - 30 дней;
- при выходе поле `revoked_at` заполняется текущим временем.

Для доступа к защищенным маршрутам пользователь должен передать токен в заголовке:

```text
Authorization: Bearer <token>
```

Серверная функция `requireAuthenticatedUser` проверяет:

- передан ли токен;
- существует ли такая сессия;
- не была ли сессия отозвана;
- не истек ли срок действия;
- существует ли пользователь, связанный с сессией.

Если проверка не проходит, API возвращает ошибку `401`.

### Валидация данных

Входные данные проверяются через Zod-схемы. Например:

- email должен быть корректным email;
- пароль должен быть не короче 8 символов;
- имя пользователя должно быть не короче 2 символов;
- score должен быть целым числом не меньше 0;
- payload тестов и материалов проверяется перед записью в базу.

Валидация снижает риск попадания некорректных данных в приложение и базу.

### Ограничение административного доступа

Административная панель защищена на двух уровнях.

Первый уровень - frontend:

- маршрут `/admin` проверяет `isLocalAdminHost`;
- если host не локальный, пользователь перенаправляется на главную;
- пункт админки не отображается в публичном меню.

Второй уровень - backend:

- все мутации контента проходят через middleware `requireLocalAccess`;
- проверяется `Origin`, `Host` или `X-Forwarded-Host`;
- разрешены `localhost`, `127.0.0.1`, `::1` и приватные IPv4-адреса;
- внешние запросы получают `403`.

Защищенные административные операции:

- `POST /api/content/tests`;
- `PUT /api/content/tests/:slug`;
- `DELETE /api/content/tests/:slug`;
- `POST /api/content/materials`;
- `PUT /api/content/materials/:slug`;
- `DELETE /api/content/materials/:slug`;
- `PUT /api/content/bundle`.

### CORS

В Express подключен CORS с настройкой origin из переменной окружения `CORS_ORIGIN`. По умолчанию разрешен frontend:

```text
http://localhost:4173
```

Это ограничивает браузерные запросы к API разрешенными источниками.

### Целостность и контроль ошибок

В проекте используется централизованная обработка ошибок Express. Если ошибка является `HttpError`, API возвращает указанный статус, иначе возвращается статус `500`.

Для операций с несколькими таблицами применяются транзакции PostgreSQL. Это особенно важно для тестов и материалов, потому что одна пользовательская операция может затрагивать сразу несколько таблиц.

### Ограничения текущей безопасности

Текущая реализация подходит для учебного проекта и локальной демонстрации. Для промышленной эксплуатации нужно добавить:

- полноценную ролевую модель администратора вместо локального доступа;
- HTTPS;
- хранение токена в HttpOnly Secure cookie либо иную усиленную модель хранения;
- защиту от brute force на маршрутах входа;
- rate limiting;
- аудит действий пользователей и администраторов;
- централизованное логирование;
- настройку секретов через защищенное окружение;
- миграции базы данных вместо прямого schema.sql;
- автоматические security-тесты;
- CSP и дополнительные HTTP security headers.

Проект называется Cyber Arena. Это русскоязычная веб-платформа для обучения кибербезопасности. Она включает интерактивные тесты и тренажеры, учебные материалы, личный профиль пользователя, рейтинг участников и локальную административную панель для управления контентом.

Архитектура клиент-серверная. Frontend написан на React 19, TypeScript и Vite. Используется React Router HashRouter, поэтому маршруты имеют вид /#/tests, /#/materials, /#/profile. Backend написан на Node.js, Express 5 и TypeScript. База данных - PostgreSQL 16. Для валидации используется Zod, для подключения к БД - pg, для запуска PostgreSQL - Docker Compose.

Основные frontend-разделы: главная страница, вход, регистрация, каталог тестов, страница теста, рейтинг, каталог материалов, страница материала, профиль, локальная админка. Основные файлы: src/app/AppRouter.tsx, src/app/AppProviders.tsx, src/hooks/useContentState.tsx, src/hooks/usePlatformState.tsx, src/hooks/useLeaderboardState.tsx, src/utils/api.ts.

Backend состоит из модулей content, users и leaderboard. Content API работает с тестами и материалами. Users API отвечает за регистрацию, вход, выход, профиль, закладки, прогресс и попытки прохождения. Leaderboard API возвращает рейтинг. Основные маршруты: /api/content/bundle, /api/content/tests, /api/content/materials, /api/users/register, /api/users/login, /api/users/current/state, /api/users/current/tests/:testSlug/attempts, /api/leaderboard/current.

PostgreSQL-модель нормализована. Основные таблицы: teams, users, user_sessions, materials, material_highlights, material_sections, material_bookmarks, material_progress, tests, test_benefits, test_deck_items, test_questions, test_question_options, test_attempts, attempt_answers. Для рейтинга используется SQL view leaderboard_current. Сущности разделены: материалы отдельно от секций, тесты отдельно от вопросов и вариантов, пользователи отдельно от попыток и прогресса. Используются внешние ключи, UUID, уникальные ограничения, enum-типы и check-ограничения.

Безопасность: пароли хешируются через crypto.scryptSync с уникальной солью; сравнение выполняется через timingSafeEqual. Авторизация основана на session token, который хранится в таблице user_sessions, действует 30 дней и передается в Authorization: Bearer <token>. Защищенные маршруты проверяют текущую сессию. Входные данные валидируются через Zod. CORS ограничивается переменной CORS_ORIGIN. Админка доступна только локально: frontend скрывает /admin на нелокальном host, backend закрывает POST/PUT/DELETE контента middleware requireLocalAccess.

Тестирование: в проекте проверены npm run lint и npm run build, обе команды проходят успешно. Unit/e2e-тесты пока не настроены. Vite предупреждает о JS chunk больше 500 kB, поэтому в будущем можно добавить code splitting.

Локальный запуск: npm install, npm run db:up, npm run dev:api, npm run dev. Frontend доступен на http://localhost:4173, API - http://localhost:4000/api, PostgreSQL - localhost:5432. Production-сборка выполняется командой npm run build, результат находится в dist.
